## Workflow Philosophy

**Prototype IS the spec. Ship working things, not documents.**

- Prefer a working prototype over a long specification document. If you can show it, don't just describe it.
- Documents exist to support decisions, not to replace demonstrations.
- Every deliverable should move us closer to something a user can interact with.
- When a document and a prototype conflict, the prototype is the source of truth.

---

## Research Principles

**Every insight must link to a source. Disconfirming evidence is mandatory.**

- Never state a user need or market fact without linking to the evidence (interview transcript, analytics screenshot, survey result, or published study).
- Actively seek disconfirming evidence for every hypothesis. If you cannot find any, you have not looked hard enough.
- Distinguish between "we observed X" (evidence) and "we believe Y" (interpretation). Label each clearly.
- When summarizing research, include at least one finding that challenges the team's current assumptions.
- Prefer direct quotes from users over paraphrased summaries.

**All phases always run, depth scales.**

- No research phase is ever skipped, even for "simple" projects.
- Depth adjusts based on project classification (see above).
- Blind spot detection runs throughout: flag gaps in real-time and at each phase boundary.

**Research methods match the questions, not defaults.**

- Don't default to interviews for every project. Use the method that best tests the hypothesis.
- Consider qualitative (interviews, observation, usability testing), quantitative (surveys, analytics, A/B tests), and secondary (competitor, market, academic) methods.
- Multiple methods strengthen findings through triangulation.

## Code Principles

_This section applies when the project includes code (prototypes, applications, etc.). For research-only projects, these principles apply to any scripts or tooling._

**Explore --> Plan --> Code. Run tests. Small commits.**

1. **Explore** - Read the relevant code and understand the existing patterns before writing anything.
2. **Plan** - Outline what you will change and why. If the change touches more than 3 files, write the plan down.
3. **Code** - Implement the change. Follow the conventions already in the codebase.
4. **Test** - Run the test suite before committing. If there are no tests for your change, write them.
5. **Commit** - Small, focused commits with clear messages. One logical change per commit.

Additional code guidelines:
- Do not introduce new dependencies without discussing it first.
- Do not refactor unrelated code in the same commit as a feature change.
- If you are unsure about an approach, create a small spike or proof-of-concept first.

## Behavioral Rules

- **When uncertain, ASK rather than assume.** If a task is ambiguous, ask a clarifying question before proceeding. It is always better to ask one question than to redo work.
- Never delete files without explicit confirmation.
- Never push to main/master without explicit confirmation.
- When you encounter a decision that could go multiple ways, document the options and recommend one with reasoning rather than just picking silently.
- If you find evidence that contradicts the current plan, surface it immediately.
- **Surface blind spots proactively.** If you notice a potential gap — an unmentioned persona, an unconsidered stakeholder, an impacted flow — flag it in the moment.
- **Accept any documentation format.** The designer may provide context as text, files, or links to external tools. Work with whatever is provided.
